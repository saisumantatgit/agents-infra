## CONFIRMED WRONGFUL PASSES

### F1 — Rate-qualifier lexical evasion (8 drafts) — PRIMARY FINDING

**Mechanism.** The round-1 fix (`numeric_ok`) only compares a rate denominator
when `_rate_qualifier` extracts one. `_RATE_QUALIFIER_RE`
(`ground_check.py:715-718`) recognizes **only** `per <word>` (Latin, whitespace-
delimited) or `/<word>`, and only within `{0,2}` words after the number. Any
other way of writing a rate makes `_rate_qualifier` return `None`. When the
claim qualifier is `None`, `numeric_ok` (`ground_check.py:887-891`) falls back to
**value+unit only** — so the bare magnitude `128000` matches the source's
`128000` and the number is certified. An 8-token verbatim span
("redis is an in memory data structure store") grounds the rest via `t1_verbatim`.
Net: a claim asserting a **per-minute / per-hour** rate (60–3600× wrong) is
certified against a **per-second** source.

**Direct mechanism probe** (`_rate_qualifier` return values):

```
'second' <= ' operations per second'      # matched (correct)
'minute' <= ' operations per minute'      # matched (correct — the round-1 fix)
None     <= ' operations each minute'
None     <= ' operations every minute'
None     <= ' operations per-minute'
None     <= ' operations a minute'
None     <= ' operations one minute'
None     <= ' operations hourly'
None     <= ' sustained disk write operations per minute'   # qualifier >2 words after number
None     <= ' operations рer minute'      # Cyrillic 'р' (see F2)
```

Every draft below returned **`gate=PASS, grounding_score=100.0,
retained_appendix=[], scored_claims=1`**, the single NUMERIC claim resolving
`GROUNDED`:

| Draft | Evasion phrasing | Why `_rate_qualifier`→None |
|---|---|---|
| `red-team-r2/rate_1_each-minute.md` | `128000 operations each minute` | "each" ≠ "per", no slash |
| `red-team-r2/rate_2_every-minute.md` | `128000 operations every minute` | "every" ≠ "per" |
| `red-team-r2/rate_3_per-hyphen-minute.md` | `128000 operations per-minute` | hyphen, no `\s+per\s+` / slash |
| `red-team-r2/rate_4_a-minute.md` | `128000 operations a minute` | "a" ≠ "per" |
| `red-team-r2/rate_7_one-minute.md` | `128000 operations one minute` | "one" ≠ "per" |
| `red-team-r2/rate_8_hourly-adverb.md` | `128000 operations hourly` | adverbial rate, no denominator word |
| `red-team-r2/rate_5_qualifier-far.md` | `128000 sustained disk write operations per minute` | 4 words before "per" exceeds the `{0,2}` window |
| `red-team-r2/rate_6_qualifier-before.md` | `at a per-minute rate of 128000 operations` | qualifier precedes the number; extractor scans only *after* it |

Observed JSON (representative, `rate_1_each-minute.md`):

```json
{"gate": "PASS", "grounding_score": 100.0, "scored_claims": 1, "vacuous": false,
 "retained_appendix": [],
 "per_claim": [{"index": 0, "kind": "NUMERIC",
   "text": "Redis is an in-memory data structure store, sustaining 128000 operations each minute [S1].",
   "verdict": "GROUNDED"}]}
```

**Responsible code:** `_RATE_QUALIFIER_RE` (`ground_check.py:715-718`) — closed
vocabulary (`per`/`/` only) and `{0,2}` proximity window; fallback at
`numeric_ok` (`ground_check.py:887-896`) treats a `None` claim qualifier as "no
rate asserted".

**Severity:** Error-B. This is not AA-MOAT-001 (that exact case, "per minute", is
now caught) — it is incomplete coverage of the *same* fix: the qualifier lexicon
and proximity window are too narrow, and the `None`→bare-match fallback is the
soft seam. **Load-bearing assumption of the round-1 fix that these break:** "a
rate the author intends is always written as `per <unit>` within two words after
the number." English rates are also written with determiners ("a/each/every/one
<unit>"), adverbs ("hourly"), hyphens ("per-minute"), and pre-posed
("per-minute rate of N").

### F2 — Rate-qualifier homoglyph bypass (1 draft)

**Draft:** `red-team-r2/unit_1_cyrillic-per.md`
`Redis is an in-memory data structure store, sustaining 128000 operations рer minute [S1].`
(the "р" in "рer" is Cyrillic U+0440, not Latin "p").

**Observed:** `gate=PASS, grounding_score=100.0, retained=0`; claim `GROUNDED`.

**Mechanism.** NFKC (correctly) does **not** fold Cyrillic→Latin, so "рer" is not
"per". `_RATE_QUALIFIER_RE` fails to match → `_rate_qualifier`→`None` → bare
value match → certified as a per-second fact. This is the F1 fallback reached via
a Unicode confusable rather than a lexical gap. The global rule to NFKC-normalize
before a safety gate does *not* defend here — the confusable survives NFKC; a
confusables/skeleton fold (or restricting the qualifier scan to a script-checked
"per") would be required.

**Responsible code:** same fallback as F1 (`ground_check.py:887-896`); the regex
at `:715-718` matches the ASCII literal `per` only.

**Severity:** Error-B. Sibling of F1; distinct defense gap (script confusable, not
vocabulary).

### F3 — Quantity-noun blindness (2 drafts) — known residual, demonstrated

**Drafts:** `red-team-r2/qty_1_gigabytes.md`, `red-team-r2/qty_2_users.md`
- `...sustaining 128000 gigabytes per second [S1].`
- `...sustaining 128000 users per second [S1].`

**Observed (both):** `gate=PASS, grounding_score=100.0, retained=0`; claim
`GROUNDED`.

**Mechanism.** `numeric_ok` compares value + magnitude-unit (percent/absolute) +
rate denominator, but **never the measured quantity noun**. Source S1 says
`128000 operations per second`; the claim says `128000 gigabytes per second` /
`128000 users per second`. Value `128000`, unit `absolute`, and rate `second` all
match, so `numeric_ok`→True; the 8-token verbatim span grounds the rest. The gate
certifies a throughput number as a data-volume / user-count number.

**Responsible code:** `_extract_numeric_mentions` (`ground_check.py:809-824`) and
`numeric_ok` (`ground_check.py:827-900`) carry no measured-quantity field.

**Severity:** Error-B. Explicitly recorded as an open residual in
`docs/open-issues/OPEN-ISSUES.md` ("numeric quantity-nouns ('operations' vs
'gigabytes') are still not compared"); reproduced concretely here for the
Phase-2b decision.

### F4 — Absence grounded by a generic corpus head-noun in a 2-query session (2 drafts)

**Drafts:** `red-team-r2/abs_1_generic-headnoun.md`,
`red-team-r2/abs_3_throughput-headnoun.md`
- `There is no benchmark for the streaming ingest workload.`
- `There is no throughput ceiling for the streaming ingest path.`

**Observed (both):** `gate=PASS, grounding_score=100.0, retained=0`; claim
`ABSENCE_SUPPORTED`.

**Mechanism.** The AA-MOAT-004 fix requires, for an entity/numeric-free subject,
that the head noun be *discriminating* — a head noun present in a strict majority
of distinct queries is rejected as a blanket corpus word
(`check_absence`, `ground_check.py:1021-1032`). But that majority filter is gated
on `len(distinct) >= 3` (`:1030`) — deliberately disabled for ≤2-query sessions
to avoid Error-A. The frozen store has exactly **2** distinct queries, both
containing "benchmark" and "throughput". A negated subject whose head noun is one
of those generic words ("no **benchmark** for the streaming ingest workload"),
and which carries **no capitalized entity and no numeric** (so `strong` anchors is
empty), falls to the head-noun path, matches both queries → `match_count = 2 ≥ 2`
→ `ABSENCE_SUPPORTED`. The store contains nothing about a "streaming ingest"
workload; the absence was never actually searched.

Observed JSON (`abs_1`):

```json
{"gate": "PASS", "grounding_score": 100.0, "scored_claims": 1, "vacuous": false,
 "retained_appendix": [],
 "per_claim": [{"index": 0, "kind": "ABSENCE",
   "text": "There is no benchmark for the streaming ingest workload.",
   "verdict": "ABSENCE_SUPPORTED"}]}
```

**Responsible code:** `check_absence` (`ground_check.py:1021-1032`) — the
`len(distinct) >= 3` guard on the discriminating-head-noun filter leaves the
2-query regime with the pre-fix generic-word collision. This is the AA-MOAT-004
mechanism surviving in small sessions, not a new root.

**Severity:** Error-B. The `>=3` carve-out is documented in the code comment as a
deliberate Error-A avoidance, but it is a live wrongful-PASS surface: many real
sessions have only 1–2 retrieval queries. **Load-bearing assumption:** "in a
2-query session, a majority-present head noun is necessarily the genuinely-
targeted subject." False — a generic corpus word ("benchmark") can be
majority-present while the specific negated subject ("streaming ingest workload")
was never searched.

### F5 — NON_CLAIM smuggle of a verbless uncited fabrication (1 draft)

**Draft:** `red-team-r2/gate_1_nonclaim-smuggle.md`
```
Redis is an in-memory data structure store used as a database and cache [S1].

Redis: unquestionably the fastest datastore in all of human history.
```

**Observed:** `gate=PASS, grounding_score=100.0, scored_claims=1, retained=0`.
The fabricated superlative is classified `NON_CLAIM` and **excluded from the
denominator**; only the one grounded claim is scored.

**Mechanism.** `_is_non_claim` (`ground_check.py:362-369`) treats a verbless
fragment carrying **no citation and no numeric** as NON_CLAIM. The moat-safe rule
correctly keeps *cited* or *numeric* verbless fragments in scope (see dead angle
"header-cited" below), but a purely rhetorical, uncited, number-free assertion
("Redis: unquestionably the fastest datastore in all of human history") is dropped
entirely and never counts against the score. Paired with one grounded claim, the
draft posts a clean PASS while containing an unsupported factual assertion.

**Responsible code:** `_is_non_claim` (`ground_check.py:360-369`) + denominator
exclusion in `score_report` (`ground_check.py:1412-1413`).

**Severity:** Error-B, but the **weakest** of the findings: the smuggled
assertion is uncited and number-free, so no *fabricated source* is certified —
the moat rule deliberately excludes such fragments to keep NON_CLAIM narrow. It
is nonetheless a draft-level wrongful PASS: a reader sees a certified document
containing an unverified superlative. Design tension worth an explicit decision
(does an uncited, verbless factual assertion belong in the denominator?).

## Recommended remediation order (evidence-anchored, for the Phase-2b decision)

1. **F1 + F2 together** (highest yield, 9 drafts, one seam). The `None`→bare-match
   fallback in `numeric_ok` is the shared soft seam. Options, in rising cost:
   broaden the qualifier lexicon (determiners `a/each/every/one/per`, adverbs
   `hourly/daily`, hyphenated `per-<unit>`), widen/relax the `{0,2}` proximity
   window, scan for a pre-posed qualifier, and add a confusables/skeleton fold (or
   ASCII-script check) so Cyrillic `рer` cannot masquerade. Preserve Error-B
   monotonicity: a claim that *states* a minute/hour rate must never match a
   per-second source. Any change here → re-run calibration → new ADR-025 CR.
2. **F4** — lower the discriminating-head-noun filter's `>=3` guard or replace the
   majority test with an absolute "generic corpus word" list scoped to
   query_provenance; must be weighed against Error-A on genuine focused 2-query
   research.
3. **F3** — capture and compare the measured quantity noun (rides with the
   Phase-2b quantity-noun decision already recorded in OPEN-ISSUES).
4. **F5** — decide whether an uncited, verbless factual assertion belongs in the
   denominator; if yes, narrow `_is_non_claim` (an ADR touching the NON_CLAIM
   boundary, given the moat-safe rationale).

All four alter the Error-A/Error-B trade-off or a gate rule → **Escalation rule #1
→ STOP and ask Sai**; these are recorded here, not patched.
